# VIX集成方案讨论文档

## 1. 需求概述

### 1.1 用户需求
在汇率/美债第一个tab页上第三个子图显示VIX恐慌指数曲线，作为市场情绪波动的重要参考指标。

### 1.2 关键要求
- 数据来源：最初提到阿里云市场API，后来改用纯FRED方案
- 展示方式：独立子图（第3行），使用折线图，紫色显示
- 更新频率：每日更新（与汇率同步）
- 历史数据：从2000年开始
- 存储方式：CSV文件存储

---

## 2. 方案选型分析

### 2.1 数据源方案对比

#### 方案A：纯FRED方案 ✅ (已选中)

| 维度 | 说明 |
|------|------|
| **数据源** | FRED API - VIXCLS序列 |
| **历史数据** | 可从1990年开始获取完整历史数据 |
| **实时数据** | 每日更新，与现有美债/汇率数据同步 |
| **API密钥** | 复用现有FRED API密钥 |
| **实现复杂度** | 低，复用现有FREDService基础设施 |
| **维护成本** | 低，无需管理额外的API服务 |
| **成本** | 无额外成本（FRED API免费） |
| **可靠性** | 高，FRED是美联储官方数据源 |

**优点**:
- 与现有架构高度一致
- 无需新增API密钥管理
- 代码实现简单，复用现有服务
- 数据质量高，更新及时

**缺点**:
- 仅包含收盘价，不包含最高价/最低价
- 数据更新可能有1-2天延迟（与美债/汇率一致）

---

#### 方案B：混合方案（FRED历史 + 阿里云实时）

| 维度 | 说明 |
|------|------|
| **数据源** | FRED API（历史）+ 阿里云API（实时） |
| **历史数据** | FRED提供2000年至今 |
| **实时数据** | 阿里云提供最近20天数据 |
| **API密钥** | 需要新增阿里云APPCODE管理 |
| **实现复杂度** | 中，需要管理两个数据源 |
| **维护成本** | 中，需要维护两套数据获取逻辑 |
| **成本** | 阿里云API可能有调用费用 |
| **可靠性** | 取决于阿里云API稳定性 |

**优点**:
- 理论上数据更新更及时

**缺点**:
- 增加系统复杂度
- 需要管理额外的API密钥
- 阿里云API历史数据获取受限
- 可能产生额外费用
- 数据一致性难以保证

---

#### 方案C：纯阿里云方案

| 维度 | 说明 |
|------|------|
| **数据源** | 阿里云市场API |
| **历史数据** | 仅最近20天（API限制） |
| **实时数据** | 实时/准实时 |
| **API密钥** | 需要新增阿里云APPCODE管理 |
| **实现复杂度** | 高，需要循环调用来获取历史数据 |
| **维护成本** | 高，需要维护独立的数据源 |
| **成本** | 阿里云API可能有调用费用 |
| **可靠性** | 取决于阿里云API稳定性 |

**优点**:
- 数据更新及时

**缺点**:
- 历史数据获取困难
- API调用次数多，成本高
- 需要新增服务模块
- 与现有架构不一致

---

**最终选择**: **方案A - 纯FRED方案**

**选择理由**:
1. 与现有架构高度一致，降低维护成本
2. 无需管理额外的API密钥
3. FRED API数据质量高且免费
4. 实现简单，开发周期短
5. 用户需求中的历史数据要求（从2000年开始）可以完全满足

**最终选择**: **方案A - 独立子图（第3行）**

## 4. 关键技术决策

### 4.1 时区处理

**决策**: 使用美国东部时区(ET)作为数据源时区，转换为UTC存储

**理由**:
- CBOE VIX指数基于美国市场
- 与美债/汇率的时区处理保持一致
- UTC作为标准存储时区，便于全球使用

**实现**:
```python
def convert_timezone(self, data: pd.Series) -> pd.Series:
    """转换时区（ET → UTC）"""
    data.index = data.index.tz_localize('US/Eastern').tz_convert('UTC')
    return data
```

---

### 4.2 并发控制

**决策**: 复用现有的全局锁机制

**理由**:
- 保持并发控制逻辑统一
- 避免并发更新导致的数据不一致
- 简化代码逻辑

**实现**:
```python
# 复用现有的锁机制
if _is_updating:
    return UpdateResponse(
        success=False,
        message="数据更新正在进行中，请稍后再试",
        error_code="UPDATE_IN_PROGRESS"
    )
```

---

### 4.3 数据对齐

**决策**: 保留所有日期，非交易日使用前向填充

**理由**:
- 用户明确要求"保留所有日期"
- 保证时间序列的连续性
- VIX在非交易日保持前一交易日的值

**实现**:
```python
# 前向填充非交易日数据
data = data.ffill()
```

---

### 4.4 错误处理

**决策**: 前端优雅降级，显示空图表

**理由**:
- 用户明确要求"优雅降级（显示空图表）"
- 不影响其他子图的正常展示
- 用户体验良好

**实现**:
```typescript
if (!vixData || vixData.length === 0) {
  return <EmptyChart message="VIX数据暂不可用" />
}
```

---

### 4.5 更新策略

**决策**: 增量更新最近7天

**理由**:
- 用户明确要求"最近7天"
- 与汇率/美债的更新策略保持一致
- 覆盖周末和节假日，确保数据完整性
