# 数据更新端点规范

## 1. 概述

### 1.1 目标
统一后端「全量初始化」与「增量更新」两类端点的实现模式，避免新增数据源时各写一套，
确保所有 update_* 端点都具备：
- 宕机恢复后不留数据洞（基于 CSV last_date 自适应增量起点）
- CSV 已是最新的短路返回（不调外部 API）

### 1.2 适用对象
- 后端开发：新增数据源 / 新增端点时按本规范实现
- 代码评审：评审 routes.py 时按本规范对照

### 1.3 适用文件
- `src/api/routes.py`：所有 fetch/update 端点的实现
- `src/services/data_service.py`：CSV 读写 + `get_last_date()` + `_compute_incremental_start()` 调用方

---

## 2. 端点命名约定

每个数据源有 2-3 个端点，路径严格遵循：

| 端点 | 用途 | 触发场景 |
|------|------|----------|
| `POST /api/fetch/{xxx}/history` | 全量初始化 | 首次部署 / 重建数据 / CSV 为空 |
| `POST /api/update/{xxx}` | 增量更新 | 日常 cadence 触发（daily / monthly） |
| `POST /api/update`（可选） | 综合更新（n8n） | n8n 定时调度，一次性更新多个数据源 |

`{xxx}` 与 `data_service.save_*` 的 key 名保持一致，例：`vix`、`tga`、`hibor`、`fund_flow`、`ted_spread`。
**严禁同一数据源出现两套命名**（如 `china-bond` vs `china_bond`）。

## 3. 全量端点规范（`/fetch/{xxx}/history`）

### 3.1 9 步骨架

```python
@router.post("/fetch/{xxx}/history", response_model=UpdateResponse)
async def fetch_{xxx}_history():
    """获取X历史数据接口 - 从 YYYY-MM-DD 开始获取全部历史数据"""
    global _is_updating

    # ① 并发锁检查
    if _is_updating:
        return UpdateResponse(
            success=False,
            message="数据更新正在进行中，请稍后再试",
            error_code="UPDATE_IN_PROGRESS",
        )
    await acquire_update_lock()

    try:
        # ② 起点日志 + 实例化
        logger.info("开始获取X历史数据...")
        service = get_xxx_service()
        data_service = get_data_service()

        # ③ 时间窗口：起点固定为 settings.historical_start_date
        latest_end = pd.Timestamp.now().normalize()
        start_date = pd.Timestamp(settings.historical_start_date)

        # ④ 拉数据
        new_data = await service.fetch_xxx(start_date, latest_end)
        if new_data.empty:
            raise Exception("未能获取到任何X数据")

        # ⑤ 清洗（仅当数据源需要时）
        # FRED 来源（VIX 等）走 convert_timezone / validate_data / normalize_data
        # HKMA / akshare / 阿里云 / FRED WTREGEN 等已是干净格式，省略此步

        # ⑥ 保存
        data_service.save_xxx(new_data)

        # ⑦ 构造响应（取 last_valid_index 当最新值）
        last_idx = new_data.last_valid_index()
        latest = XxxData(
            date=last_idx.date(),
            value=float(new_data[last_idx]),
        )
        response_data = XxxUpdateData(xxx=latest)

        # ⑧ 成功返回
        logger.info("X历史数据获取成功")
        return UpdateResponse(
            success=True,
            message="X历史数据获取成功",
            data=response_data,
            updated_at=datetime.now().isoformat(),
        )

    except Exception as e:
        logger.error(f"获取X历史数据失败: {str(e)}")
        return UpdateResponse(
            success=False,
            message=f"获取X历史数据失败: {str(e)}",
            error_code="UPDATE_FAILED",
        )
    finally:
        # ⑨ 释放锁（必走，不论成败）
        release_update_lock()
```

### 3.2 时间窗口起点

**永远**用 `settings.historical_start_date`（默认 `2000-01-01`）。**禁止**硬编码日期字符串。
特例：
- OECD 债券（`/update` 综合端点）：`latest_end - 365 days`（频率低）
- 资金流向（fund-flow）：`settings.fund_flow_start_date = '2014-11-17'`（北向资金开通日）

## 4. 增量端点规范（`/update/{xxx}`）

### 4.1 5 步骨架

```python
@router.post("/update/{xxx}", response_model=UpdateResponse)
async def update_{xxx}():
    """增量更新X数据 - 从 CSV last_date+1 拉到今天"""
    global _is_updating

    # ① 并发锁检查（同 §3.1 ①）
    if _is_updating:
        return UpdateResponse(success=False, message="数据更新正在进行中，请稍后再试", error_code="UPDATE_IN_PROGRESS")
    await acquire_update_lock()

    try:
        logger.info("开始增量更新X数据...")
        service = get_xxx_service()
        data_service = get_data_service()

        latest_end = pd.Timestamp.now().normalize()
        # ② 增量起点：必须用 _compute_incremental_start，禁硬编码 today-N days
        start_date = _compute_incremental_start(data_service, "{xxx}", latest_end)

        # ③ 已是最新短路（见 §5）
        if start_date is None or start_date > latest_end:
            logger.info("X数据已是最新，无需更新")
            return UpdateResponse(
                success=True,
                message="X数据已是最新，无需更新",
                data=XxxUpdateData(xxx=XxxData(date=latest_end.date(), value=None)),
                updated_at=datetime.now().isoformat(),
            )

        # ④ 拉数据 + 保存（参照 §3.1 ④-⑦）
        logger.info(f"增量更新X数据，从 {start_date} 到 {latest_end}")
        new_data = await service.fetch_xxx(start_date, latest_end)
        if new_data.empty:
            raise Exception("未能获取到任何X新数据")
        data_service.save_xxx(new_data)
        last_idx = new_data.last_valid_index()
        response_data = XxxUpdateData(xxx=XxxData(date=last_idx.date(), value=float(new_data[last_idx])))
        logger.info("X数据增量更新成功")
        return UpdateResponse(
            success=True,
            message="X数据增量更新成功",
            data=response_data,
            updated_at=datetime.now().isoformat(),
        )

    except Exception as e:
        logger.error(f"X数据增量更新失败: {str(e)}")
        return UpdateResponse(success=False, message=f"X数据增量更新失败: {str(e)}", error_code="UPDATE_FAILED")
    finally:
        release_update_lock()
```

### 4.2 增量起点策略（**强制**）

**必须**用 `_compute_incremental_start(data_service, "{xxx}", latest_end)`，**禁止**硬编码 `today - N days`。

**为什么**：硬编码 today-7 days 在宕机超 7 天恢复时会留数据洞（CSV last_date 之后有空缺日期）。

`_compute_incremental_start` 行为（见 `routes.py` 第 185 行）：
- CSV 为空 → 返回 `settings.historical_start_date`（全量）
- CSV 已有数据 → 返回 `last_date + 1 day`（补齐 gap）
- start_date 已超过今天 → 返回 `None`（已是最新）

## 5. 「已是最新」短路（**强制**）

每个 `update_*` 端点**必须**在 `start_date = _compute_incremental_start(...)` 之后立刻检查：

```python
if start_date is None or start_date > latest_end:
    logger.info("X数据已是最新，无需更新")
    return UpdateResponse(
        success=True,
        message="X数据已是最新，无需更新",
        data=XxxUpdateData(xxx=XxxData(date=latest_end.date(), value=None)),  # value=None 占位
        updated_at=datetime.now().isoformat(),
    )
```

**为什么**：避免每次 cadence 触发都去调外部 API（FRED / HKMA / 阿里云），节省配额 + 降低失败率。

注意：`start_date > latest_end` 兜底对应 last_date 推到今天的情况（理论上 `_compute_incremental_start` 已经处理，但保留双保险对齐 china_bonds 实现）。

## 6. 清洗三步（仅 VIX 适用）

`convert_timezone` / `validate_data` / `normalize_data` 只在 VIX 数据上调用，因为：

| 数据源 | 清洗需求 |
|--------|----------|
| VIX（VIXCLS） | FRED 返回含时区 Series → 需 convert_timezone / validate / normalize |
| TGA（WTREGEN） | FRED WTREGEN 已是干净数字 → **省略** |
| HIBOR（HKMA） | HKMA API 返回 ISO 字符串 → **省略** |
| fund-flow（akshare） | akshare 已是干净 DataFrame → **省略** |
| ted-spread（FRED SOFR + DGS3MO） | FRED 已是干净 Series → **省略** |
| china_bonds | 内部 service 已处理 → **省略** |
| commodities（comkm） | service 内已处理 → **省略** |
| indices（comkm） | service 内已处理 → **省略** |

新增数据源时**必须**评估是否需要这三步；如需要，按 `update_vix` 实现。

## 7. 响应 model 约定

`UpdateResponse` 顶层（见 `models.py`）：

```python
class UpdateResponse(BaseModel):
    success: bool
    message: str
    error_code: Optional[str] = None
    data: Optional[BaseModel] = None  # 各数据源自己的 XxxUpdateData
    updated_at: Optional[str] = None  # ISO 字符串
```

`error_code` 枚举（**禁止新增**未列出的值）：

| error_code | 含义 |
|------------|------|
| `UPDATE_IN_PROGRESS` | 并发占用（另一 update 还在跑） |
| `UPDATE_FAILED` | 拉数据失败 / 保存失败 / 任何异常 |

## 8. 日志规范

每个端点固定 3-4 条日志：

```
开始获取X历史数据...
获取X历史数据，从 {start} 到 {end}
X历史数据获取成功 / 获取X历史数据失败：{err}
```

增量端点：

```
开始增量更新X数据...
增量更新X数据，从 {start} 到 {end}      # 短路时换成：X数据已是最新，无需更新
X数据增量更新成功 / X数据增量更新失败：{err}
```
